# Rime 雙欄式【候選清單】架構與除錯總結

本文件定義「雙欄式候選清單」之格式。

雙欄【候選清單】，
- 左欄：輸入方案使用的【漢字標音方法/系統】，如：
    - 拼音輸入法：台語音標/白話字/台羅拼音/閩拼方案/台語注音二式
    - 注音輸入法：方音符號/台語注音符號，如：
    - 反切輸入法：十五音，如：君二柳


## 一、 預期目標與架構分工

如以下之雙欄並列的【候選清單】格式，其産生過程式由「**YAML 轉換** 與 **Lua 腳本**」分工原則：

`經【ㄍㆥ】〔kenn1〕`

為：左欄的內容，
1. **左欄【方音符號】**
   - 負責元件：Rime【輸入方案】檔案（.schema.yaml）中的**YAML 設定檔**（如 `lib_comment_tps_and_tlpa.yaml`）。
   - 處理邏輯：透過正規表達式（`xform`），將原始字典的台語拼音轉換為對應的方音符號 `ㄍㆥ`。

2. **右欄〔台語音標〕**
   - 負責元件：**YAML 設定檔**。
   - 處理邏輯：純粹顯示字典裡的原始拼音字串 `kenn1`，僅為對照與參考用途，**不作任何進一步轉換**。

3. **Lua 腳本的角色 (`rime.lua`)**
   - **容許介入的情境**：僅針對「十五音」（包含 `huan_ciat` 反切 或 `hau_suan` 校算）的特定方案，因為這些方案需要對聲韻調做內部漢字倒裝（聲韻調 -> 韻調聲）。
   - **嚴格禁止的情境**：對於如 `zu_im_tlpa` 或 `zu_im_tps` 等非十五音的拼音/注音方案，Lua 必須 **完全迴避**，不得攔截或竄改 YAML 輸出的 comment 字串。

---

## 二、 遇到的問題與錯誤現象

在開發過程中，原本的輸出結果為錯誤的 `經【ㄍkⁿe】〔Lㆤ〕` 或出現 `〔kⁿe〕` 等亂碼及錯置情況。經除錯分析，發現兩大雷區：

### 1. YAML 正規表達式（Regex）定錨點（Anchor）失準
在 `lib_comment_tps_and_tlpa.yaml` 處理鼻化韻母（如 `kenn` → `Lㆤ` → `ㆥ`）時，由於標記處理進度的定錨點符號（`》`）位置擺放錯誤（例如應寫成 `Lㆤ》` 卻寫錯對齊位置），導致正規表達式無法正確捕捉並置換鼻化韻，產出了 `Lㆤ` 這種殘缺的中介字串。

### 2. Lua 腳本的過度介入（越權轉換與倒裝）
這也是導致右欄拼音被竄改的最根本原因。在原先的 `rime.lua` 之中：
- `reformat_comment_filter` 與內部的 `format_comment` 沒有對非十五音方案進行豁免。
- Lua 攔截了 YAML 原本處理好的格式，擅自載入 `tlpa_converter.lua` 模組，將純英數字的拼音（`kenn1`）強制轉成了帶有上標符號的格式（如 `keⁿ1` / `kⁿe`，顯示上類似 `⁰`）。
- Lua 同時還將 `〔...〕` 與 `【...】` 的包裹標記進行倒裝重組，從而讓左右欄呈現的內容完全亂位。

---

## 三、 解決方案與程式碼修正

針對上述問題，我們透過以下修正，確立了最終版正確的資料流向：

### 1. 修正 YAML 鼻音轉換規則
調整 `lib_comment_tps_and_tlpa.yaml` 中所有鼻音對應的 `xform` 表達式。將左欄 `【】` 內的字首加上進度錨點：
```yaml
- xform/^(.*)$/【》$1】〔$1〕/
```
確認後續的 `xform` 置換（如將 `kenn` 換為 `ㆥ`）能精準抓到目標，使左欄順利產出如 `【ㄍㆥ】`的注音。

### 2. 閹割（Bypass） Lua 對常規方案的干擾
在 `rime.lua` 的 `format_comment` 函數中，加上了最關鍵的「**非十五音方案豁免機制**」：

```lua
local function format_comment(comment_string, mode, schema_id)
    if type(comment_string) ~= "string" or comment_string == "" then
            return comment_string
    end

    local is_sni = schema_id and (schema_id:match("hau_suan") or schema_id:match("huan_ciat"))

    -- 【關鍵修正】：對於非十五音（如 tps, tlpa），完全由 yaml 控制左右欄，直接回傳不作變更
    if not is_sni then
            return comment_string
    end

    -- (下方保留十五音方案專用的倒裝與格式邏輯...)
end
```

## 四、 結論

透過排除 Lua 在常規輸入方案（TLPA/TPS）中的作用，我們讓雙欄格式徹底交由 Rime 的 YAML `xform` 引擎全權接管。如此一來：
- 不會再發生 `kenn1` 被偷偷蓋換成 `keⁿ1`。
- 括號 `【】` 與 `〔〕` 的相對位置能完美保持原樣。
- 從資料流的一致性與日後維護的角度來看，這確保了底層純拼音字典不受 Lua 應用層渲染邏輯的干擾，達到最健壯的運作狀態。


## 設計規格

### 反切輸入法

基於【字典】所用【漢字標音法】為【台語音標】或【台語注音二式】，其【音節】的結構為：【聲母】+【韻母】+【調號】。

以【漢字】：忍為例，其【台語音標】為：lun2 = l + un + 2，故推導出：

- 聲母（聲）：柳
- 韻母（韻）：君
- 調號（調）：二

但【十五音】的【音節】結構卻為：【韻母】+【調號】+【聲母】。

- 韻母（韻）：君
- 調號（調）：二
- 聲母（聲）：柳

也就是說，【輸入方案】在【輸入編輯列】(preedit_fromat) 處理來自鍵盤的按鍵輸入，需配合【字典】使用的【漢字標音法】，因而有【聲、韻、調】順序的先天限制，如此一來，只能用【柳君二】來完成輸入。

但是標準的【十五音】其【漢字標音法】則應使用【君二柳】，遵循【韻、調、聲】的【音節】順序。本專案的【反切輸入法】的解決方案為：不在【輸入編輯列】進行輸入調整，而是在【候選清單】顯示校正回歸的正確結果。

其作法大致為：（1）在輸入方案（.schema.yaml）檔案中的 yaml script ，負責産出【候選清單】於【左欄】顯示的【十五音】，其【音節】格式為：【聲韻調】（同【輸入編輯列】）；然後，透過【輸入方案】使用的 Lua Script 函數 reformat_comment_filter() ，負責將【候選清單】左欄中的【音節】標音，將格式轉換成【韻調聲】格式。
